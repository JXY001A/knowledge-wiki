> 通过文件系统解耦 System Prompt 与动态内容，将稳定的系统指令与易变的业务逻辑分层管理，Agent 按需加载上下文的工程方法论。

## 是什么

上下文工程是 Agent Prompt 从“深耦合”向“渐进式加载”演进过程中形成的方法论。核心理念是：固化 System Prompt（保留最稳定的底层指令和行为规范），将动态的、具体的任务要求、领域知识、人设规范等剥离出来，存储到外部文件系统（如 SKILL.md、USER.md、CLAUDE.md），通过渐进式披露按需加载。

## 为什么重要

- **解决 Prompt 膨胀**：早期一个任务一个 Agent，System Prompt 内卷到数千字，维护混乱
- **System Prompt 稳定**：固化后不再频繁修改，降低回归风险
- **动态内容模块化**：业务逻辑变更只需修改对应文件，不影响核心行为
- **上下文窗口高效利用**：只加载当前任务需要的内容，减少 token 浪费
- **团队协作**：不同文件由不同角色维护（如产品经理维护 USER.md，工程师维护 SKILL.md）

## 工作原理

1. **分层设计**：System Prompt（稳定层）→ 用户配置（USER.md/SOUL.md）→ 技能文件（SKILL.md）→ 领域知识（wiki/*.md）
2. **渐进式披露**：任务启动时只加载稳定层，根据任务意图逐层加载后续文件
3. **文件系统即注册表**：目录结构本身就是 Skill 的发现机制
4. **Git 原生版本管理**：所有内容变更可追溯、可回滚

## 与相邻概念的区别

- **vs Prompt Engineering**：Prompt Engineering 关注单条指令的优化；上下文工程关注整体内容的分层架构
- **vs RAG**：RAG 是检索外部知识后注入上下文；上下文工程是预设的结构化分层加载
- **vs 知识工程**：知识工程更关注知识的组织和提取；上下文工程关注 Agent 运行时的内容加载策略
