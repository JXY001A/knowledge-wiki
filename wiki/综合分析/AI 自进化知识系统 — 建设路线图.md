> 以当前知识库基础设施为基座，按五层架构分阶段建设具备自进化能力的 AI Agent 系统。最终目标是让 Agent 能够沉淀经验、发现知识缺口、自动优化自身行为，实现“越用越好用”。

## 一、现状基线

### 已建成

| 能力 | 实现 | 成熟度 |
|------|------|--------|
| 知识摄入入口 | 企业微信 Bot（文本存入、URL 下载、? 查询） | 可用，但质量待提升 |
| 三层数据架构 | `raw/` → `wiki/` → Schema（`AGENTS.md`） | 成熟 |
| MCP 服务 | `wiki-mcp-server`：ingest / query / search / lint | 成熟 |
| 多端访问 | 企业微信（手机）+ Claude Code（Mac）+ Obsidian（桌面） | 成熟 |
| Git 同步 | Mac ↔ GitHub ↔ DevMechin 自动双向同步 | 成熟 |
| 本地推理 | Ollama（qwen2.5:3b, qwen3-vl:8b）+ RTX 4080 16GB | 可用，但模型太小 |
| Agent Skill 概念 | Agent Skill 规范、渐进式披露、5 种设计模式已文档化 | 概念层，未工程化 |

### 核心缺口

1. **ingest 质量低** — qwen2.5:3b 无法按 `AGENTS.md` 规范生成高质量 wiki 页面
2. **无记忆系统** — 每次查询独立，不沉淀经验
3. **无评估机制** — 不知道回答好不好、知识覆盖全不全
4. **无自动进化** — 不能从错误中学习、不能主动补全知识

## 二、参考框架：Agent 四阶段演进

来自参考文章《Agent技术范式演变》：

```
阶段一：被动式 ReAct (2023)
  LLM + Planning + Tools + Memory
  “一问一答”，单步推理

阶段二：Workflow Agent (2024)
  工程化约束弥补模型不确定性
  LangGraph / Dify 流程编排

阶段三：自主 Agent (2025)
  复杂 Planning + 长程任务 + 自我校验
  Claude Code / Codex / OpenClaw

阶段四：自进化 Agent (2026)
  沉淀经验 → 优化策略 → 自动升级
  “越用越好用”，从消耗品变为可积累资产
```

我们当前处于**阶段一到阶段二之间**：有基础的 ReAct 模式（企业微信 Bot 的 `? query → MCP → LLM → 回答`），但缺乏 Workflow 的结构化约束和自主 Agent 的 Planning 能力。

## 三、五个核心维度的演化方向

文章拆解的六个技术维度中，与我们最直接相关的五个：

| 维度 | 早期做法 | 现在方向 | 我们的目标 |
|------|---------|---------|-----------|
| **Prompt** | 单体大 System Prompt | 固化 System Prompt + 渐进式加载外部 Skill 文件 | `AGENTS.md` 做稳定底座，Skill 按需加载 |
| **Memory** | 向量 DB 主导 | 文件系统回归 + 事项型/知识型双轨 | `wiki/` 做语义记忆，`操作日志` 升级为情景记忆 |
| **Tools** | 预定义 Function Call | MCP 动态工具发现 | 已有 `wiki-mcp-server`，持续扩展 |
| **Planning** | 单步 CoT | 复杂任务拆解 + 多步协同 + 子 Agent | 引入 Planning 引擎做任务路由 |
| **Workflow** | 无 | 轻量 Harness + 自我校验 | 评估反馈闭环 |

## 四、五层目标架构

```
Layer 5: 自进化引擎

        └─────────────────────────────────────────────┘
                          ↑
Layer 4: 评估与反馈
        ┌─────────────────────────────────────────────┐
        │  回答准确性评分 │ 知识覆盖率仪表盘 │ 用户反馈收集   │
        │  错误案例自动记录 │ lint 增强（语义+事实检查）  │
        └─────────────────────────────────────────────┘
                          ↑
Layer 3: 记忆系统
        ┌─────────────────────────────────────────────┐
        │  工作记忆（会话窗口）                         │
        │  情景记忆（结构化操作日志 + 经验库）            │

        └─────────────────────────────────────────────┘
                          ↑

        ┌─────────────────────────────────────────────┐
        │  SKILL.md 渐进式加载 │ MCP 工具注册与发现      │

        └─────────────────────────────────────────────┘

Layer 1: 知识基座  ← 当前位置
        ┌─────────────────────────────────────────────┐

        │  wiki/ 知识网络 + 概念关联 + 交叉引用          │
        │  MCP 检索服务 + 多端查询入口                  │
        └─────────────────────────────────────────────┘
```

## 五、分阶段实施计划

### Phase 1：知识基座升级（1-2 周）

**目标：ingest 质量达到 Claude Code 水平，知识库成为可信赖的单一事实源。**

| 任务 | 要点 | 优先级 |
|------|------|--------|
| 1.1 DeepSeek API 替代本地小模型 | `_call_llm_ingest` 从 Ollama 切换为 DeepSeek API；system prompt 嵌入完整 `AGENTS.md` + `templates/source.md`；128K 上下文直接处理完整文章 | P0 |
| 1.2 多 URL 类型兼容 | 微信公众号、GitHub、技术博客、视频字幕等不同来源的内容清洗策略 | P1 |
| 1.3 概念自动提取与关联 | 识别新概念 → 自动创建 `wiki/概念/` 页面 → 双向 [[wikilinks]] 关联 | P1 |
| 1.4 图表自动生成 | 文章包含架构/流程/数据流时自动生成 Excalidraw（后续 Phase 再启用） | P2 |
| 1.5 ingest 结果通知优化 | 企业微信推送摘要卡片 + 链接到 Obsidian 查看 | P2 |

**交付标准**：微信发一篇中等复杂度的技术文章 → 30s 内生成完整 wiki 页面（资料摘要 + 概念页 + 交叉引用 + 目录更新 + 操作日志），质量接近 Claude Code 手动 ingest。

**关键决策**：
- 模型选择：DeepSeek API（`deepseek-chat`，¥0.01/篇，128K 上下文足够嵌入完整 ingest 规范）
- 本地模型保留用于 query（低延迟、免费），DeepSeek 用于 ingest（高质量、按量付费）

---

### Phase 2：Agent 技能体系（2-3 周）

**目标：知识不再只是被“查”，而是被 Agent“执行”。**

| 任务 | 要点 | 优先级 |
|------|------|--------|
| 2.1 SKILL.md 规范设计 | 定义 Skill 文件格式（触发条件、输入/输出 schema、执行步骤、前置依赖、评估标准） | P0 |
| 2.2 渐进式加载引擎 | Agent 启动加载 `AGENTS.md`（System Prompt），执行任务时按需加载对应 `SKILL.md`；文件系统目录即技能注册表 | P0 |
| 2.3 首批核心 Skill | `ingest-article`、`query-knowledge`、`lint-wiki`、`create-concept` 等 5-8 个 Skill | P1 |
| 2.4 Skill 版本管理 | Git 追踪 Skill 变更历史，支持回滚和效果对比 | P2 |
| 2.5 多模型路由 | 简单 query → 本地 qwen2.5:3b（免费低延迟）；复杂 ingest/分析 → DeepSeek API；图像理解 → qwen3-vl:8b | P1 |

**交付标准**：MCP 增加 `skill-list`、`skill-execute` 工具；企业微信 `?` 查询从硬编码逻辑改为 Skill 驱动。

**架构示意**：
```
用户: "? 帮我整理一篇关于 Transformer 的文章"
  → Planning 引擎
    → 识别意图: "ingest-article"
    → 加载 SKILL.md: ingest-article
    → 执行步骤: download → defuddle → DeepSeek 分析 → 创建 wiki 页面 → 更新目录
    → 报告结果
```

---

### Phase 3：记忆系统（2-3 周）

**目标：每次交互都沉淀为可检索、可复用的经验。**

| 任务 | 要点 | 优先级 |
|------|------|--------|
| 3.1 工作记忆 | 会话上下文管理：当前对话的关键信息提取、上下文窗口压缩 | P0 |
| 3.2 情景记忆升级 | `操作日志.md` 从纯文本日志升级为结构化经验库：每次 query/ingest 记录 → 结果评分 → 经验标签 → 可检索 | P0 |
| 3.3 用户画像 | `USER.md`：用户偏好、常用领域、知识水平、反馈历史 | P1 |
| 3.4 语义记忆增强 | `wiki/` 概念关联图自动化：检测孤页、死链、概念覆盖度，通过 Canvas 可视化 | P1 |
| 3.5 经验回放 | 定期回顾历史交互中的失败案例，提取改进点，沉淀到 Skill 或知识页面 | P2 |

**关键设计决策（来自参考文章）**：
- 长期记忆以**文件系统为主**，而非向量数据库 —— 与当前 `wiki/` 架构一致
- 区分**事项型记忆**（任务执行记录）和**知识型记忆**（概念理解）—— 分别存入情景记忆和语义记忆

**数据结构设计**：
```
wiki/记忆/
├── 情景/
│   ├── query-20260522-001.md   # 每次查询的结构化记录
│   └── ingest-20260522-001.md  # 每次摄取的结构化记录
├── 用户/
│   └── USER.md                  # 用户画像
└── 经验/
    ├── 什么是好的ingest.md       # 从历史中提炼的经验
    └── 常见知识缺口.md
```

---

### Phase 4：评估与反馈（2-3 周）

**目标：系统知道自己做得好不好，并能向用户报告。**

| 任务 | 要点 | 优先级 |
|------|------|--------|
| 4.1 回答质量评分 | 每次 query 后自动评估：准确性（是否基于 wiki 事实）、完整性（是否覆盖所有相关知识点）、有用性（是否解决了用户问题） | P0 |
| 4.2 知识覆盖率仪表盘 | 检测 wiki 覆盖盲区：哪些领域文章少、哪些概念缺独立页面、哪些 raw 资料未处理 | P1 |
| 4.3 用户反馈收集 | 企业微信消息的下方添加 👍👎 反馈按钮（template_card 支持）；反馈记录关联到具体 query | P1 |
| 4.4 lint 增强 | 从格式检查扩展到语义检查：事实矛盾检测、信息过时提醒、置信度不一致 | P2 |
| 4.5 自动错误案例记录 | 低评分 query 自动保存为案例，包含问题、回答、评分、改进建议 | P2 |

**评分机制**：

评估由 DeepSeek API 执行（低成本高质量判断）：

```
输入：用户问题 + wiki 检索结果 + 生成的回答
输出：{
  accuracy: 1-5,    // 基于 wiki 事实的准确性
  completeness: 1-5, // 知识覆盖的完整度
  usefulness: 1-5,   // 对用户的有用程度
  gaps: ["缺 X 概念页", "Y 领域资料不足"],
  improvement: "建议..."
}
```

---

### Phase 5：自进化闭环（3-4 周）

**目标：系统自动发现知识缺口、自动补全、自动优化行为。**

| 任务 | 要点 | 优先级 |
|------|------|--------|
| 5.1 知识缺口自动检测 | 分析多次低分 query 的共性 → 识别知识盲区 → 生成待摄取清单 | P0 |
| 5.2 自动 ingest 触发 | 检测到缺口 → 自动搜索相关 URL → 推送给用户确认 → 自动 ingest | P0 |
| 5.3 Skill 效果追踪 | 每个 Skill 的执行成功率、平均评分、耗时 → 自动标记“需优化”Skill | P1 |
| 5.4 Prompt 自动优化 | A/B 测试不同 System Prompt 变体 → 收集评分 → 自动采用更优版本 | P2 |
| 5.5 定期自检报告 | 每周自动生成“系统健康报告”：新增知识量、查询准确率趋势、Top 知识缺口、Skill 表现排行 | P1 |
| 5.6 模型微调实验 | 积累足够高质量 QA 对后 → LoRA 微调本地模型提升 query 质量（长期探索） | P3 |

**自进化循环**：
```
          ┌──────────────────────┐
          │   用户使用（query）    │
          └──────────┬───────────┘
                     ↓
          ┌──────────────────────┐
          │   评估（评分 + 反馈）  │
          └──────────┬───────────┘
                     ↓
          ┌──────────────────────┐
          │   分析（缺口 + 根因）  │
          └──────────┬───────────┘
                     ↓
          ┌──────────────────────┐
          │   行动（ingest / 优化）│
          └──────────┬───────────┘
                     ↓
          ┌──────────────────────┐
          │   验证（评分回升）     │
          └──────────────────────┘
```

## 六、技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| **ingest 模型** | DeepSeek API (`deepseek-chat`) | 128K 上下文，中文优秀，¥0.01/篇，指令遵循接近 Claude |
| **query 模型** | 本地 qwen2.5:3b（短期）→ DeepSeek（升级路径） | 低延迟免费，后续可按需升级 |
| **评估模型** | DeepSeek API | 需要高质量判断，本地小模型不足以胜任 |
| **知识存储** | Git + Markdown（沿用） | 文章核心观点：文件系统是长期记忆最佳载体 |
| **工具协议** | MCP（沿用） | 动态工具发现，渐进式扩展 |
| **自动化引擎** | Python + systemd timer + MCP | 轻量，无额外依赖 |

## 七、关键设计原则

来自参考文章，作为所有 Phase 的约束：

1. **不是替代关系** — 四个阶段并存互补。Phase 2 不推翻 Phase 1，而是在其上叠加
2. **文件系统是长期记忆最好的载体** — 不引入向量数据库，延续 `wiki/*.md` + git 的模式
3. **System Prompt 固化 + Skill 外置** — 延续 `AGENTS.md` + `templates/` 的渐进式披露架构
4. **Harness 要轻** — 用最少的工程约束获得最大的可控性，避免过度编排
5. **先质量后数量** — 知识库的质量决定上层所有能力的天花板，Phase 1 是最高优先级

## 九、风险与依赖

| 风险 | 影响 | 缓解 |
|------|------|------|
| DeepSeek API 不稳定 | Phase 1 受阻 | 保留本地 qwen2.5:3b 做降级 |
| 本地模型无法胜任 query | Phase 4 评估不准确 | 评估用 DeepSeek API |
| 知识库规模增长失控 | wiki/ 页面过多难以维护 | lint 增强 + 自动归档 |
| 自进化走偏 | 越优化越差 | 所有自动化保留人类审核节点 |
