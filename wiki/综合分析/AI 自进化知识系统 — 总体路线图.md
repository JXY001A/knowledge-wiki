> **本文是 AI 自进化知识系统的唯一权威路线图。** 所有其他规划文档的交叉引用最终汇总于此。分为一条主进化路线（Phase 1-5）和三条功能增强线。本文取代分散在多个文档中的独立规划。

## 一、全景图

```
                            AI 自进化知识系统
                                  │
        ┌─────────────────────────┼─────────────────────────┐
        │                         │                         │
        ▼                         ▼                         ▼
  ┌──────────┐            ┌──────────────┐          ┌──────────────┐
  │  主进化路线 │            │  功能增强线    │          │  运维基线     │
  │  (垂直攀登) │            │  (水平扩展)    │          │  (持续运行)   │
  └──────────┘            └──────────────┘          └──────────────┘
        │                         │                         │
        │                         │                         │
  Phase 1 ██ 知识基座       Track A ██ 检索升级      部署拓扑
  Phase 2 ██ 技能体系       Track B ██ 个人助手      Git 同步
  Phase 3 ██ 记忆系统       Track C ░░ 图表自动化     健康监控
  Phase 4 ██ 评估反馈
  Phase 5 ░░ 自进化闭环

  ██ = 已完成    ░░ = 待开始    ▓▓ = 进行中
```

### 线与线的关系

| 关系 | 说明 |
|------|------|
| **主路线是前提** | Phase 1-2 的完成使功能增强线成为可能。没有知识基座和技能引擎，检索升级和助手扩展无处附着 |
| **增强线反哺主路线** | Track A（检索升级）直接提升 Phase 3（记忆系统）的检索质量；Track B（个人助手）是 Phase 3 的落地形式之一 |
| **各自独立推进** | 三条线可并行开发，不互相阻塞。但都依赖主路线 Phase 2 的技能引擎 |

### Phase 1：知识基座 `✅ 完成于 2026-05-23`

> **目标**：ingest 质量达到生产级，知识库成为可信赖的单一事实源。

### Phase 3：记忆系统 `✅ 完成于 2026-06-07（提前 19 天）`

> **目标**：每次交互都沉淀为可检索、可复用的经验。从"一问一答"进化为"有记忆的助手"。

| # | 任务 | 状态 | 实现 |
|---|------|:---:|------|
| 3.1 | **工作记忆** | ✅ | `memory/working.py` — 会话上下文提取 + LLM 注入 |
| 3.2 | **情景记忆升级** | ✅ | `memory/writer.py` — SQLite + 文件双写；`reader.py` — 结构化查询 + FTS5 全文搜索 |
| 3.3 | **用户画像** | ✅ | `memory/profile.py` — `wiki/USER.md` 自动生成与维护 |
| 3.4 | **语义记忆增强** | ✅ | `memory/semantic.py` — 概念覆盖度分析（缺失概念/关联密度/领域分布） |
| 3.5 | **经验回放** | ⏳ | 留待 Phase 4 评估系统完成后 |

**关键设计决策**：
- 长期记忆以**文件系统为主**，不引入向量数据库 —— 与现有 `wiki/` 架构一致
- 区分**事项型记忆**（任务执行记录）和**知识型记忆**（概念理解）
- 操作日志升级为每行 JSON 的结构化记录，而非自然语言段落

**当前进度**：只有基本操作日志（272 行纯文本），无结构化、无可检索性、无评分机制。

| # | 任务 | 状态 | 实现 |

| 4.2 | **知识覆盖率仪表盘** | ✅ | `memory/semantic.py` + `mcp/tools/lint.py` 增强 — 概念覆盖度、领域分布、评估统计 |
| 4.3 | **用户反馈收集** | ⏳ | 留待企微 template_card 集成 |
| 4.4 | **lint 语义增强** | ✅ | lint 工具增加评估统计和记忆统计展示 |
| 4.5 | **自动错误案例记录** | ⏳ | 低评分自动记录到 memory_events（Phase 5 闭环用） |

**评估由 DeepSeek API 执行**（低成本高质量判断）：

```
输入：用户问题 + wiki 检索结果 + 生成的回答
输出：{
  accuracy: 1-5,       // 基于 wiki 事实的准确性
  completeness: 1-5,   // 知识覆盖的完整度
  usefulness: 1-5,     // 对用户的有用程度
  gaps: ["缺 X 概念页", "Y 领域资料不足"],
  improvement: "建议..."
}
```

### Phase 5：自进化闭环 `⏳ 待开始（路线图计划 08-07 启动）`

> **目标**：系统自动发现知识缺口、自动补全、自动优化行为。从"人在驱动"变为"系统自驱"。

| # | 任务 | 说明 | 优先级 |
|---|------|------|:---:|
| 5.1 | **知识缺口自动检测** | 分析多次低分 query 的共性 → 识别知识盲区 → 生成待摄取清单 | P0 |
| 5.2 | **自动 ingest 触发** | 检测到缺口 → 自动搜索相关 URL → 推送给用户确认 → 自动 ingest | P0 |
| 5.3 | **Skill 效果追踪** | 每个 Skill 的执行成功率、平均评分、耗时 → 自动标记"需优化"Skill | P1 |
| 5.4 | **Prompt 自动优化** | A/B 测试不同 System Prompt 变体 → 收集评分 → 自动采用更优版本 | P2 |
| 5.5 | **定期自检报告** | 每周自动生成系统健康报告：新增知识量、查询准确率趋势、Top 知识缺口、Skill 表现 | P1 |
| 5.6 | **模型微调实验** | 积累足够高质量 QA 对后 → LoRA 微调本地模型提升 query 质量（长期探索） | P3 |

**自进化循环**：

```
          ┌──────────────────────┐
          │   用户使用（query）    │  ← 日常使用
          └──────────┬───────────┘
                     ↓
          ┌──────────────────────┐
          │   评估（评分 + 反馈）  │  ← Phase 4
          └──────────┬───────────┘
                     ↓
          ┌──────────────────────┐
          │   分析（缺口 + 根因）  │  ← Phase 5
          └──────────┬───────────┘
                     ↓
          ┌──────────────────────┐
          │   行动（ingest/优化）  │  ← Phase 5
          └──────────┬───────────┘
                     ↓
          ┌──────────────────────┐
          │   验证（评分回升）     │  ← Phase 4 回环
          └──────────────────────┘
```

**关键约束**：所有自动化操作保留人类审核节点（"检测到 X 可能填补知识缺口，是否摄取？"），防止自进化走偏。

### Track A：检索系统升级 `✅ 完成于 2026-05-30`

> **目标**：从"关键词 grep"升级为"语义理解 + 结构化分层"，使 LLM 获得高质量上下文。

```
升级前：keyword in text 全量扫描（O(n)，精度依赖关键词精确匹配）

升级后：catalog 初筛 → BM25 多字段加权 → layered 分层组装
         Phase 0           Phase 1          Phase 2        Phase 3
          预处理           目录过滤          BM25排序       分层输出
```

| 阶段 | 模块 | 功能 |
|------|------|------|
| Phase 0 | `tokenizer.py` | jieba 分词 + 去停用词 + 搜索模式 |
| Phase 1 | `catalog.py` | 解析 Wiki 目录.md → 标题命中 + 类别 boost → Top-K=20 初筛 |
| Phase 2 | `bm25.py` | BM25 Okapi 多字段加权（title×3.0 / tags×2.0 / body×1.0）排序 |
| Phase 3 | `layered.py` | L1 编译真相（Top-1 + 400字摘录）→ L2 交叉引用（wikilink 摘要）→ L3 证据补充（Top 2-N 全文），Token 预算 8000 |
| — | `index_store.py` | pickle + zstd 压缩序列化，版本号 + 文档数 + 抽样 hash 三重校验 |
| 留待 | 图谱加权 | 200 页后启用 PageRank 语义关联加权 |

**效果**：检索精度从关键词盲扫提升到 BM25 结构化排序，LLM 上下文从 800 字符片段提升到 8000 token 分层结构化 context。

### Track B：个人 AI 助手扩展 `📋 方案完成于 2026-06-06`

> **目标**：从"知识库"扩展为"个人 AI 助理"——待办、提醒、笔记、书签、习惯追踪等日常能力。

**核心设计**：

| 方面 | 选择 |
|------|------|
| 存储 | 文件系统（知识内容）+ SQLite（结构化助理数据）双层架构 |
| 扩展方式 | 以 Skill 形式注入（skill.json + SKILL.md + impl.py），不改动已有模块 |
| 入口 | 企微 Bot（移动端）+ MCP Server（AI CLI 工具），共享同一套数据 |
| 推送 | 从被动回复突破为主动推送（调度器 + 企微 API） |

**新增内容**：

| 类型 | 数量 | 详情 |
|------|:---:|------|
| 新技能 | 7 个 | todo-manage / remind-set / note-quick / bookmark-save / schedule-view / daily-brief / habit-track |
| MCP 工具 | 14 个 | todo_list / todo_create / todo_complete / remind_set / remind_list / note_create / note_search / bookmark_add / bookmark_list / daily_brief / schedule_today / habit_create / habit_log / habit_stats |
| 数据库表 | 7 张 | todos / reminders / notes / bookmarks / habits / habit_logs / push_queue |
| Python 模块 | 8 个 | db.py / scheduler.py / backup.py / push.py / models/（todo/reminder/note/bookmark/habit） |

**关联主路线**：Track B 是 Phase 3（记忆系统）的落地形式——待办和提醒是情景记忆，笔记是语义记忆，习惯追踪是经验数据。

## 七、关键设计原则

来自 Karpathy LLM Wiki + Agent Skill 设计模式，贯穿所有 Phase：

1. **不是替代，是叠加** — 每个 Phase 不推翻前一个，而是在其上增强。Phase 3 不重写 Phase 2 的技能引擎，只是增加记忆层
2. **文件系统是长期记忆最好的载体** — 不引入向量数据库，wiki/*.md + SQLite（仅辅助索引）+ Git 版本控制
3. **System Prompt 固化 + Skill 外置** — AGENTS.md 做稳定底座，Skill 渐进式加载，互不污染
4. **Harness 要轻** — 用最少的工程约束获得最大的可控性，避免过度编排
5. **先质量后数量** — 知识库的质量决定上层所有能力的天花板
6. **人类审核节点不可省略** — 自动化 ingest、自动优化 Prompt 等自进化操作，必须经过人类确认
7. **增量扩展，零破坏** — 新能力以 Skill 形式注入，不改动已有模块代码

## 八、技术选型总表

| 组件 | 选型 | 角色 |
|------|------|------|
| **ingest 模型** | DeepSeek API (`deepseek-v4-pro`) | 高质量内容分析与 wiki 页面生成 |
| **query 模型** | Ollama 本地 (`qwen2.5:3b`) | 低延迟免费的知识问答 |
| **评估模型** | DeepSeek API | 需要高质量判断，本地小模型不胜任 |
| **视觉模型** | Ollama 本地 (`qwen3-vl:8b`) | MCP Vision Server，GPU 加速 |
| **知识存储** | Markdown 文件 + Git | 人类可读，版本可追踪，Obsidian 可浏览 |
| **助理数据** | SQLite | 结构化查询，高频写入，辅助索引 |
| **检索引擎** | BM25 Okapi + jieba 分词 | 中文友好的多字段加权检索 |
| **工具协议** | MCP (Model Context Protocol) | 动态工具发现，标准 AI 互操作 |
| **消息接入** | 企业微信 Bot | 移动端入口，URL 摄取 + 知识查询 |
| **调度引擎** | APScheduler | 定时任务（早报/备份/提醒） |
| **部署** | systemd + FRP | DevMechin 7×24 常驻，外部可达 |
