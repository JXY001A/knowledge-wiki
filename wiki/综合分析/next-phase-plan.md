## Phase 6：RAG 2.0 — 回答质量升级

### 问题

当前 query 管线是 `BM25 关键词检索 → qwen2.5:3b 生成回答`。两个致命缺陷：

1. **检索只认字不认义**：用户搜索"怎么让 AI 自己变强"，BM25 搜的是"AI""变强"这些字，不会返回"自进化"相关的 wiki 页面（虽然标题和正文里没有出现"变强"这个词）
2. **小模型答不好**：qwen2.5:3b 只有 3B 参数，对需要综合多个 wiki 页面的复杂问题，回答质量低。评估均分 3.0/5

### 方案

```
当前：
  用户问题 → BM25 关键词检索 → qwen2.5:3b 生成 → 回答

升级后：
  用户问题 → 双路检索 → 重排序 → DeepSeek API 生成 → 带引用回答
              │
              ├─ BM25（关键词快速召回 Top-20）
              └─ Embedding（语义向量召回 Top-20）
              └─ 合并去重 → RRF 融合 → Top-10
```

### 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| **Embedding 模型** | `bge-large-zh-v1.5` (本地) 或 DeepSeek Embedding API | 中文语义理解好，本地 CPU 可跑 |
| **向量存储** | `sqlite-vec` 扩展 或 `faiss` | 零运维，与现有 SQLite 统一 |
| **重排序** | `bge-reranker-large` (本地 CPU) | 部署成本最低 |
| **回答模型** | DeepSeek API (`deepseek-chat`) | 已经接入，只需切换到 query 场景 |

### 回答格式

```markdown
## 自进化 AI 的核心原理

自进化 Agent 是指系统能够通过反馈回路持续优化自身行为……
（来源：[[自进化Agent]]）

在工程实践中，Harness 模式通过外部约束引导进化方向……
（来源：[[Harness Engineering]]）

> 📚 参考：[[AI 自进化知识系统 — 总体路线图]] | [[Agent 技术范式演变]]
> 🎯 综合评分：A4/C4/U4（准确性 4/5，完整度 4/5，有用性 4/5）
```

### 实施

1. **embedding 索引**：为所有 wiki 页面生成向量，增量更新（ingest 时自动向量化）
2. **双路检索**：`retrieval/embedding.py` — 语义搜索；`retrieval/fusion.py` — RRF 融合
3. **query 模型切换**：简单问题（≤200 字答案）用 Ollama，复杂问题走 DeepSeek
4. **引用生成**：LLM 输出时要求标注来源 `[[页面名]]`

|------|:---:|:---:|

## Phase 7：多轮对话引擎

### 问题

Web Chat 已存储历史到 SQLite，但 `process_message` 每次独立处理。用户体验：

```
用户: 什么是 MCP 协议？
AI: （回答 MCP）

用户: 和 REST API 比哪个好？
AI: （不知道"哪个"指什么）❌
```

### 方案

```
用户消息 → 注入最近 N 轮对话历史 → 意图识别 → 检索 + 生成 → 存储本轮

对话历史格式：
[
  {"role": "user", "content": "什么是 MCP？", "time": "09:30"},
  {"role": "bot", "content": "MCP 是 Model Context Protocol...", "time": "09:30"},
  {"role": "user", "content": "和 REST API 比哪个好？", "time": "09:31"},
]
```

### 技术要点

**1. 上下文窗口管理**

```python
# Token 预算：8000（检索结果 4000 + 对话历史 2000 + 回答 2000）
def build_context(conv_id: str, question: str, wiki_context: str) -> str:
    history = get_recent_messages(conv_id, max_tokens=2000)
    return f"""## 对话历史
{history}

## 知识库检索结果
{wiki_context}

## 当前问题
{question}"""
```

**2. 指代消解**

将"那个""它""上面的"替换为具体引用。简单方案：把上一轮用户问题追加到当前问题：

```python
if has_pronouns(question) and history:
    question = f"（上文：用户问了「{history[-2]['content']}」）{question}"
```

**3. 会话摘要**

超过 20 轮时自动压缩历史为摘要：

```python
if len(history) > 20:
    summary = call_ollama(f"用 100 字总结这段对话：{history}")
    history = [{"role": "system", "content": f"对话摘要：{summary}"}] + history[-5:]
```

### 接入点

- `webhook/process.py` 的 `chat_api`：读历史 → 注入上下文 → 保存新消息
- Chat.tsx：无改动（历史通过 `/chat/convs` API 加载）

## Phase 8：主动智能

### 8.1 智能推送

```
当前推送：定时触发（早上 8:00 早报 / 下午 18:00 晚报）
                        ↑ 不管有没有事都推

升级后：定时触发 + 事件触发 + 上下文感知
        早上 8:00 早报  →  有到期待办时才推（避免噪音）
       提醒前 30min    →  预推送："10:00 的会议还有 30 分钟"
       晚上 21:00      →  如果今天没打卡习惯，推送"今天还没打卡哦"
```

### 8.2 自动提醒升级

当前提醒用的是 `threading.Timer`（webhook 进程内），进程重启丢失。

升级：APScheduler job 持久化到 SQLite，重启后自动恢复。

### 8.3 日报/周报自动生成

```
每天晚上 21:00：
  - 今天完成的待办
  - 今天新增的笔记
  - 今天的习惯打卡情况
  - 今天新增的 wiki 页面
  → 生成日报草稿 → 推送 → 用户确认后保存到 wiki/日报/

每周日 22:00：
  - 本周 wiki 增长统计
  - 本周评估趋势
  - 下周待办汇总
  → 生成周报 → 推送
```

## Phase 9：语音交互

### 9.1 语音转文字（ASR）

硬件已就绪（USB 麦克风+音箱），缺少 ASR 链路。

| 方案 | 优点 | 缺点 |
|------|------|------|
| OpenAI Whisper API | 识别率最高 | 要钱、延迟略高 |
| `whisper.cpp` 本地 | 免费、低延迟 | 中文略弱、需 GPU |
| 企微语音消息 | 已接入、免费 | 60s 限制 |

**推荐**：先接企微语音消息（零成本），再部署 `whisper.cpp` 做本地 ASR。

### 9.2 文字转语音（TTS）

已有 `speak-text` Skill（edge-tts），但每次需 2-3 秒生成 mp3。升级方向：

- **流式 TTS**：文本生成和语音播放并行（减少等待）
- **Web 端**：浏览器 SpeechSynthesis API（零成本）

### 9.3 语音对话循环

```
用户语音 → ASR → 文字 → 检索+生成 → 回答文本 → TTS → 音箱播放
                                                        ↓
                                              同时推送到企微
```

## Phase 10：知识深加工

### 10.1 语义关联网络

**问题**：wiki 页面间只有手动 wikilink，缺少自动发现的关系。

**方案**：对所有 wiki 页面做 embedding → 计算余弦相似度 → 发现潜在关联：

```
[[自进化Agent]] ← 0.87 → [[Agent Skills体系]]  （强关联，已有 wikilink）
[[自进化Agent]] ← 0.72 → [[上下文工程]]        （中等关联，建议添加 wikilink）
[[自进化Agent]] ← 0.65 → [[Harness Engineering]] （中等关联，建议添加）
```

输出为"关系发现建议"，人类审核后自动添加 wikilink。

### 10.2 矛盾检测增强

当前 lint 有基本矛盾检测框架，但只靠规则匹配（关键词冲突）。升级为 LLM 驱动：

```python
# 将两篇页面内容送给 DeepSeek
prompt = f"""
页面 A：{page_a_content}
页面 B：{page_b_content}
是否存在事实性矛盾？仅在有明确矛盾时报告。
"""
```

## 技术选型总表（新增）

| 组件 | 选型 | 用量估算 |
|------|------|------|
| Embedding | `bge-large-zh-v1.5` 本地 | CPU，~1.3GB 内存 |

```

  Phase 7: 多轮对话 — Chat 上下文注入
```

## 不改的

- 不引入 LangChain/LlamaIndex（太重，自己写 200 行够用）
- 不换向量数据库（FAISS 内存方案对 1000 页以下完全够）
- 不拆微服务（Flask 单体继续，当前规模无需）
- 不改已有 Skill 接口（向下兼容）

## 风险

| 风险 | 应对 |
|------|------|
| DeepSeek API 做 query 成本高 | 简单问题回退 Ollama，分级调用 |
| embedding 模型吃内存 | bge-small 替代（~400MB），排名掉 5% 可接受 |
| 企微语音 60s 限制 | 长语音切分 + 逐段处理 |
