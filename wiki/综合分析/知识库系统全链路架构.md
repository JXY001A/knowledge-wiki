## 二、模型安装与选型

### Ollama 运行时

DevMechin 的 Ollama 安装流程：

1. **下载二进制**：从 GitHub Releases 获取 `ollama-linux-amd64.tar.zst`
   - 直连 GitHub 下载慢时，通过国内镜像（`ghproxy.net`、`gh.con.sh`）加速
   - 如 Mac 有代理，通过 `curl -x socks5://127.0.0.1:1087` 代理下载后 scp 到 DevMechin
2. **安装**：`sudo tar --zstd -xf /tmp/ollama-linux.tar.zst -C /usr`，将 `ollama` 和 `ollama.service` 放到系统路径
3. **启动**：`sudo systemctl start ollama`，绑定 `127.0.0.1:11434`
4. **拉取模型**：`ollama pull <model-name>`

安装参考：[Ollama Linux 安装文档](https://ollama.com/download/linux)

### 模型清单

| 模型 | 部署位置 | 用途 | 大小 |
|------|----------|------|------|
| `qwen2.5:1.5b` | DevMechin | 企业微信 RAG 回答（主） | ~1GB |
| `llama3.2:1b` | DevMechin | 企业微信 RAG 回答（回退） | ~1GB |
| `qwen3-vl:8b` | DevMechin | 图片解析（`~/bin/vision`） | ~5GB |
| LLaVA:7b | Mac | 本地图片解析（回退） | ~4GB |

**模型选型约束**：

- 企业微信场景要求低延迟（用户等待 <30s），因此选 1.5B 级小模型
- `qwen2.5:1.5b` 中文理解和 JSON 输出能力优于同类尺寸模型
- `llama3.2:1b` 作为回退，当 qwen 空响应或超时时自动切换
- `qwen3-vl:8b` 用于图片解析，8-dimension UI 分析 prompt，GPU 加速
- Mac Ollama 版本 v0.7.0 过旧，不支持 Qwen3-VL，待升级到 v0.12.7+

## 三、Wiki MCP 服务

### 服务端（DevMechin）

**实现**: Python FastMCP + Streamable HTTP 传输

**路径**: `~/code/wiki-mcp-server/server.py`

**systemd 单元**（`~/.config/systemd/user/wiki-mcp.service`）：

```ini

After=network.target

[Service]
ExecStart=/usr/bin/python3 /home/jxy001a1/code/wiki-mcp-server/server.py
Restart=always
RestartSec=5

[Install]
WantedBy=default.target
```

**四个 MCP 工具**：

| 工具 | 输入 | 行为 |
|------|------|------|
| `ingest(path or URL)` | 文件路径或 URL | `git pull --rebase` → 按 Schema 生成/更新 wiki 页面 → `git add -A && git commit && git push` |
| `query(question)` | 自然语言问题 | `git pull --rebase` → 读 wiki 页面 → 综合回答（附 `[[wikilink]]` 引用） |
| `lint()` | 无 | `git pull --rebase` → 巡检（孤页/死链/过时/矛盾/标签规范/资料覆盖/概念覆盖） |
| `search(keyword)` | 关键词 | `grep wiki/` → 返回匹配页面列表 |

**同步机制**: 每次操作前 `git pull --rebase` 拉取 Mac 端的最新变更，操作后 `git push` 推送。个人使用场景下冲突窗口极小，无需额外消息队列。

## 四、企业微信 Bot 接入

### 完整调用链

```
手机微信
  │ 用户发消息
  ▼
企业微信 Bot（回调 URL: http://8.133.175.201:9400/webhook）
  │ POST XML（AES 加密）
  ▼
阿里云 frps → DevMechin frpc → 127.0.0.1:9400
  │
  ▼
wecom-webhook.py（Flask，829 行）
  │
  ├─ 文本消息 → 保存到 raw/收件箱/ → git push
  │
  └─ ?开头查询 → RAG 检索 wiki/ → 调用 Ollama 小模型 → template_card 多卡片回复
```

### 消息加解密

企业微信回调使用 AES-256-CBC 加密 + SHA1 签名验证：

1. **URL 验证**（`GET /webhook`）：验签 → AES 解密 `echostr` → 返回明文
2. **消息接收**（`POST /webhook`）：验签 → AES 解密 XML → PKCS7 unpadding → 提取 `FromUserName`、`Content`
3. **环境变量**: `WECOM_TOKEN`、`WECOM_AES_KEY`（Base64）、`WECOM_CORP_ID`

### 消息处理流程（`_process`）

```
收到消息
  │
  ├─ 以 ? 开头 ──► 查询模式：_handle_query(user_id, question)
  │                  │
  │                  ├─ 关键字检索 wiki/*.md（路径匹配权重 ×2）
  │                  ├─ top-5 结果排序
  │                  │
  │                  ├─ 单 section 页面（无 ## 或只有 1 个）
  │                  │   └─ _call_llm_json() → JSON → _send_template_card()
  │                  │       {cards: [{title, summary, details:[[k,v],...]}]}
  │                  │
  │                  └─ 多 section 页面（≥2 个 ## 标题）
  │                      └─ 每 section 一张 template_card
  │                          ├─ _call_llm_short() 生成每节摘要
  │                          └─ 800ms 间隔依次发送（防止企业微信乱序）
  │
  └─ 其他文本 ──► 保存模式：_save_to_inbox() → _git_push()
```

### 主动消息 API

通过企业微信 API 发送消息，需要先获取 access_token（缓存 7200s）：

**Markdown 消息**（`_send_markdown`）：

```json
{
  "touser": "<user_id>",
  "msgtype": "markdown",
  "agentid": 1000002,
  "markdown": {"content": "..."}
}
```

**模板卡片消息**（`_send_template_card`）：

```json
{
  "touser": "<user_id>",
  "msgtype": "template_card",
  "agentid": 1000002,
  "template_card": {
    "card_type": "text_notice",
    "source": {"desc": "知识库", "desc_color": 0},
    "main_title": {"title": "...", "desc": "..."},
    "horizontal_content_list": [
      {"keyname": "k1", "value": "v1"}
    ],
    "card_action": {"type": 1, "url": "https://work.weixin.qq.com"}
  }
}
```

**多卡片发送顺序保证**：每个模板卡片之间 sleep 800ms。企业微信不保证顺序送达，必须通过间隔避免乱序。

### 查询检索机制

```python
# 关键字权重策略
score = sum(1 for kw in keywords if kw in body.lower())          # 正文匹配 ×1
score += sum(2 for kw in keywords if kw in path.lower())         # 路径匹配 ×2
```

简单但有效：路径名中的关键词匹配权重加倍，优先推荐标题相关的页面。

## 五、小模型内容输出约束

企业微信 Bot 的核心挑战是：**用 1.5B 级小模型在 30s 内产出格式合规、内容准确的回答**。为此设计了一套严格的 prompt 约束和容错机制。

### 5.1 模型选择与回退

```
qwen2.5:1.5b（主）
  ├─ num_predict: 1200-1500
  ├─ temperature: 0.1-0.3（JSON 用 0.1 确保格式稳定）
  ├─ timeout: 60-120s
  │
  └─ 失败/空响应时 → llama3.2:1b（回退）
       ├─ num_predict: 200
       ├─ temperature: 0.1
       ├─ context 截断至 1500 字符
       └─ 仍然失败 → 直接返回 wiki 原文前 800 字符
```

### 5.2 Markdown 输出约束

`_call_llm_detailed()` 的 system prompt 规定了严格的输出格式：

```
- 用中文回答，详细但不过于啰嗦
- 用 ## 标题分段
- 用 **粗体** 强调关键术语
- 用 `代码` 标记技术名词
- 用 - 列表展示多项信息
- 不要用表格（|...|），不要用代码块（```）
- 只基于提供的资料，不要编造
- 200-400字
```

**禁止表格和代码块的原因**：企业微信 Markdown 消息不支持表格渲染和围栏代码块，会原样显示 `|...|` 和 ` ``` ` 造成阅读混乱。

**清洗函数** `_clean_markdown()`:

```python
text = re.sub(r"```[\s\S]*?```", lambda m: "[代码]", text)  # 代码块→标记
text = re.sub(r"<[^>]+>", "", text)                          # 去 HTML
# 过滤表格行（以 | 开头且包含 |...| 模式的行）
if re.match(r"^[\s|:\-]+$", s): continue
```

### 5.3 JSON 结构化输出约束

`_call_llm_json()` 要求小模型输出严格 JSON，用于生成 `template_card`：

```json
{
  "cards": [
    {
      "title": "分类名称",
      "summary": "100-200字摘要",
      "details": [["键1", "值1"], ["键2", "值2"]]
    }
  ]
}
```

**Prompt 约束**：
- `"Output ONLY valid JSON. No other text."`
- 分成 2-4 个逻辑分类
- 每个卡片 3-8 个 key-value 对
- 只用真实数据，不编造占位符
- `num_predict: 1200`, `temperature: 0.1`（低温度提高格式稳定度）

**JSON 修复容错**（小模型容易产出格式瑕疵）：

```python
# 1. 去除 markdown 代码块标记
content_clean = re.sub(r"```(?:json)?\s*", "", content_clean)
# 2. 括号匹配提取 JSON
# 3. 直接 parse 失败后，修复常见错误：
fixed = re.sub(r",\s*([}\]])", r"\1", json_str)  # 尾逗号
fixed = re.sub(r"([{,])\s*([a-zA-Z_]+)\s*:", r'\1"\2":', fixed)  # 未引号 key
# 4. 仍失败则 fallback 到 section_kv 提取
```

### 5.4 Template Card 格式化

`_normalize_details()` 容错处理多种 LLM 输出格式：

```python
# 支持三种格式：
# 1. [[k, v], [k, v]]        → 标准格式
# 2. [{k: v}, {k: v}]        → dict 格式
# 3. [k, v, k, v, ...]       → 平铺列表格式
# 最多取 10 对
```

### 5.5 多 Section 卡片策略

对于包含多个 `##` 标题的 wiki 页面：

```
_split_by_sections(body, page_title)
  │
  ├─ 过滤噪声 section（"参考资料", "See Also", "外部链接"）
  ├─ 最多 8 个 section
  │
  └─ 每个 section：
       ├─ _call_llm_short() 用 llama3.2:1b 生成一句话摘要
       ├─ _extract_kv_from_markdown() 提取表格 key-value
       ├─ _send_template_card() 发送
       └─ time.sleep(0.8)  ← 800ms 间隔保证企业微信不重排
```

每个卡片附 `source_desc` 标注来源页面和序号（如 "DevMechin（AI 主机） (1/3)"）。

### 5.6 各 LLM 调用函数对照

| 函数 | 模型 | 用途 | num_predict | temp |
|------|------|------|-------------|------|
| `_call_llm_json` | qwen2.5:1.5b | 生成 template_card JSON | 1200 | 0.1 |
| `_call_llm_detailed` | qwen2.5:1.5b | 生成 markdown 详细回答 | 1500 | 0.3 |
| `_call_llm_short` | llama3.2:1b | section 一句话摘要 | 80 | 0.1 |
| `_call_llm` | llama3.2:1b | RAG markdown 回退 | 600 | 0.1 |
| `_call_llm_fallback` | llama3.2:1b | 最终回退 | 200 | 0.1 |

### 5.7 调试与可观测性

- LLM JSON 原始输出保存至 `/tmp/llm_debug.txt`
- 关键路径有 `print(..., flush=True)` 日志
- `/health` 端点返回配置状态和环境信息
