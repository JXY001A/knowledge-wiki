> 将原本分散在 3 个独立 repo 中的代码合并为统一 Python 项目，并实现 Phase 2 Agent 技能体系。核心变化：从“内容仓库 + 散落脚本”进化为“完整项目系统”，企业微信消息分发从硬编码改为 Skill 驱动。

## 关键论点

### 1. 从三体到单体：合并即架构

**改造前：**

| 代码位置 | 形式 | 问题 |
|---------|------|------|
| `~/code/wiki-mcp-server/server.py` (DevMechin) | 单文件 392 行 | 无模块拆分，独立 repo |
| `~/code/wecom-bot-webhook/webhook.py` (DevMechin) | 单文件 1148 行 | 逻辑高度耦合，独立 repo |
| `.claude/wecom-webhook.py` (Mac) | 过期副本 | 与 DevMechin 不同步 |

**改造后：** 统一为 `knowledge-wiki` 下的 `src/knowledge_wiki/` Python 包，28 个模块文件，2066 行。

- `wiki/` — 共享操作：git、frontmatter、搜索、页面构建、日志（从两处重复代码提取合并）
- `llm/` — LLM 客户端：DeepSeek API + Ollama（统一接口、JSON 修复容错）
- `mcp/` — MCP 服务端 + 客户端 + 4 个工具模块 + Phase 2 的 `skill-list` / `skill-execute`
- `webhook/` — 企业微信 Bot：Flask 路由、Skill 分发、wechat 集成
- `skill/` — 技能执行引擎：注册表、渐进式加载、意图匹配、多模型路由
- `app/` — 应用入口：CLI 解析、server 启动、webhook 启动
- `skills/` — 文件系统技能注册表：5 个核心技能（skill.json + SKILL.md + impl.py）

### 2. 模块依赖图

```
config.py（pydantic-settings，单例）
    │
    ├── wiki/ (git, frontmatter, search, builder, log)
    │     └── mcp/tools/ (ingest, query, lint, search)
    │           └── app/server.py → python -m knowledge_wiki serve
    │
    ├── llm/ (deepseek, ollama)
    │     │
    │     └── skills/*/impl.py  ← 技能实现调用 LLM
    │           │
    │           └── skill/planner.py  → execute_skill()
    │                 │
    │                 └── webhook/process.py  → process_message()
    │                       │
    │                       └── app/webhook.py → python -m knowledge_wiki webhook
    │
    ├── mcp/client.py ← webhook/skills 调用本地 MCP 工具
    │
    └── skill/ (registry, engine, planner, router)
          ├── mcp/server.py → skill-list / skill-execute 工具
          └── skills/ (文件系统注册表，5 个技能)
```

### 2.1 Phase 2 技能架构

技能执行流程：

```
企业微信消息
  │
  ▼
process_message(user_id, text, send_md, send_tpl)
  │
  ├─ ?text → query="text" → match_skill(query)
  ├─ URL   → match_skill(url)
  └─ text  → match_skill(text) → fallback: save-note
  │
  ▼
execute_skill(name, context)
  │
  ├─ 1. find_skill(name) → Skill 对象
  ├─ 2. _load_impl(skill) → import skills/<name>/impl.py
  └─ 3. impl.execute(context) → 结果
       │
       ├─ search-wiki   → wiki.search.search_wiki()
       ├─ query-knowledge → mcp.client.mcp_query() + llm.ollama.call_json()
       ├─ ingest-article  → fetch_url_text() + llm.deepseek.call_ingest()
       ├─ lint-wiki       → mcp.tools.lint.lint_tool()
       └─ save-note       → wiki.paths.save_to_inbox()
```

五个核心技能的触发关系：

```
用户输入
  │
  ├─ 包含 URL ────────────────────────► ingest-article (deepseek)
  │
  ├─ ? + "搜索/查找/有哪些" ───────────► search-wiki (local)
  ├─ ? + "是什么/怎么/为什么/介绍" ────► query-knowledge (local)
  ├─ "检查/lint/巡检" ────────────────► lint-wiki (local)
  │
  └─ 其他文本 ────────────────────────► save-note (local，默认回退)
```

### 3. 与五层架构的映射

| 架构层 | 对应模块 | 状态 |
|--------|---------|------|
| Layer 1: 知识基座 | `wiki/` + `llm/deepseek.py` + `mcp/tools/` | 已工程化 |
| Layer 2: Agent 技能体系 | `skill/` + `skills/`（5 核心技能） | **已实现** |
| Layer 3: 记忆系统 | `wiki/log.py` → 后续升级为结构化经验库 | 预留接口 |
| Layer 4: 评估与反馈 | — | 后续 Phase |
| Layer 5: 自进化引擎 | — | 后续 Phase |

### 3.1 Layer 2 技能清单

| 技能 | Tier | 模型 | 触发词 | 核心调用 |
|------|------|------|--------|---------|
| `search-wiki` | 1 | local | 搜索、查找、search、find | `wiki.search.search_wiki()` |
| `query-knowledge` | 1 | local | 查询、是什么、怎么、为什么 | `mcp.client.mcp_query()` → `llm.ollama.call_json()` |
| `ingest-article` | 1 | deepseek | 摄取、ingest、URL | `fetch_url_text()` → `llm.deepseek.call_ingest()` |
| `save-note` | 1 | local | 保存、记录、笔记 | `wiki.paths.save_to_inbox()`（默认回退） |
| `lint-wiki` | 2 | local | 检查、健康、lint、巡检 | `mcp.tools.lint.lint_tool()` |

### 4. 关键设计决策

**`src/` 布局。** Python 代码全部放在 `src/knowledge_wiki/` 下，与 `raw/`、`wiki/`、`templates/` 等 Obsidian vault 内容彻底分离。`pyproject.toml` 在根目录，`[tool.setuptools.packages.find]` 指向 `src/`。

**单 venv、双服务。** MCP Server 和 Webhook 共享同一个 Python 环境。`kw-server serve` 和 `kw-server webhook` 两个 CLI 子命令分别启动，systemd 单元指向同一个 `.venv`。

**文件系统即注册表。** `skills/` 目录结构本身就是技能注册表。每个技能一个子目录，`skill.json` 定义元数据，`SKILL.md` 定义完整行为。无需数据库或额外注册文件——延续“文件系统是最好的长期记忆”原则。

**渐进式加载三级。** Tier 1（启动时，name+description，~50 tokens/技能）→ Tier 2（意图匹配后，SKILL.md 全文）→ Tier 3（执行需要时，scripts/references）。20 个技能仅消耗 ~1000 tokens 即可纳入 System Prompt。

**环境变量集中管理。** `config.py` 使用 pydantic-settings 从 `.env` 文件加载所有配置，systemd 通过 `EnvironmentFile=` 引用。敏感信息不再散落在 systemd 文件中，`.env` 被 gitignore 排除。

**安全改进。** 旧的 wecom-webhook.service 在 `Environment=` 中以明文暴露 `DEEPSEEK_API_KEY`、`WECOM_SECRET` 等密钥。新部署将它们移入 gitignored `.env` 文件。

## 证据

- **测试覆盖**：`tests/` 目录 32 项单元测试全部通过（wiki: 8, llm: 9, webhook: 3, skill: 12）
- **CLI 可用**：`python -m knowledge_wiki --help` 正常输出 `{serve,webhook}` 子命令
- **导入验证**：所有 7 个包（wiki, llm, mcp, webhook, skill, app, config）的导入链完整
- **代码量**：28 个源模块约 2200 行；5 个技能实现（skill.json + SKILL.md + impl.py）约 700 行
- **技能注册表验证**：`list_skills()` 返回 5 个技能，意图匹配准确率 7/8
- **技能执行验证**：`search-wiki` 搜索返回 8 条匹配结果，`lint-wiki` 巡检发现 15 条死链
- **Deploy 脚本**：`deploy/setup.sh` 实现 DevMechin 一键部署（git pull → venv → systemd 迁移）

## 开放问题

1. **Intent 匹配精度**：当前基于关键词的意图匹配（`match_skill`）对 URL 类输入需硬编码回退。后续可引入 LLM 意图识别提升准确率
2. **Skill 热加载**：当前 `_load_impl()` 通过 `importlib` 首次加载后缓存到 `sys.modules`，修改技能需重启服务。后续可加入文件监听自动重载
3. **Skill A/B 测试**（Phase 2 P2）：Git 追踪 Skill 变更历史，支持版本回滚和效果对比
4. **query-knowledge 依赖回调**：技能通过 `send_md` / `send_tpl` 回调发送消息，`execute_skill` 返回值仅作日志。独立的非 webhook 调用场景（如 CLI/MCP）需额外适配

## 启示

**工程骨架决定上限，但真正的价值在“执行”而非“描述”**。Phase 2 从“技能是什么”的文档阶段，进化为“技能会执行”的工程阶段——5 个 `impl.py` 各自调用现有的 `wiki/`、`llm/`、`mcp/` 模块，webhook 的 `process_message` 从 15 行硬编码三路分发变成 30 行 Skill 驱动的路由引擎。

这验证了 Phase 1 重构的核心假设：模块化的 `wiki/` 和 `llm/` 基础层使得技能实现可以“一行代码调用现有能力”，而不是在每个技能中重复实现。**好的分层让上层变薄，坏的分层让上层越来越厚。**
