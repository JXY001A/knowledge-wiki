> 本文系统梳理了2023至2026年Agent核心技术概念的演变逻辑，从宏观四个阶段（被动ReAct→工作流→自主→自进化）切入，深入剖析Prompt、Planning、Memory、Tools、Workflow、Environment六大维度的范式迁移。揭示了Agent从“单体指令”到“系统工程”的转变，强调通过工程化手段承载模型不确定性，实现能力跃升的内核重构，为技术选型与落地提供思想框架。

## 核心要点

- Agent经历被动ReAct、工作流、自主规划、自进化四个阶段，能力逐级跃升
- Prompt从紧耦合的单体System Prompt转向渐进式加载的上下文工程
- Planning由线性思维链升级为长程任务拆解、协同执行与自我校验
- 记忆模块从向量主导转向文件系统化沉淀与向量检索混合管理
- 工具调用从API封装演变为利用模型原生CLI与脚本执行，降低开发维护成本
- 工作流与Skill混合封装，刚性编排与动态能力结合保证稳定性

## 详细笔记

## Agent发展的四个阶段

### 阶段一：早期Agent（被动式ReAct）
- 时间：2023年，LLM爆发元年
- 代表性理论：Lilian Weng的《LLM Powered Autonomous Agents》，架构为LLM + Planning + Tools + Memory
- 核心模式：基于ReAct（Reasoning + Acting），单步“Reasoning → Observe → Response”
- 特点：类似增强版Chatbot，一问一答，严重依赖明确指令，只能完成短链路小任务，缺乏长期规划，超出上下文窗口易中断
- 代表性项目：AgentGPT、AutoGen、MetaGPT

### 阶段二：工作流Agent（结构化与可控性）
- 时间：2024年
- 背景：纯ReAct无法满足to B稳定性要求
- 核心理念：用工程化约束弥补模型不确定性，引入硬约束和流程编排（Harness早期形态）
- 架构特征：固定Workflow中嵌入LLM节点，或LLM调用预定义子Workflow；牺牲灵活性换取高可控性和可解释性
- 应用场景：企业服务、日常重复性工作，按步骤保质完成
- 价值：非长尾、非极度复杂的场景性价比最高，落地最稳定，至今大量企业使用

### 阶段三：自主Agent（复杂规划与长程任务）
- 时间：2025年关键转折点
- 代表产品：Manus、Claude Code、Codex等
- 核心变化：具备复杂Planning能力，可自行拆解任务、规划路径、调用工具、多轮迭代
- 长程任务能力：给定需求和开发规范（Specs），Agent可长时间自主处理企业级代码或复杂业务流程
- 自我校验：配合轻量级Harness或自我校验机制，长程运行中修正错误
- 角色转变：从“辅助者”变为“执行者”

### 阶段四：自进化Agent（持续学习与自我升级）
- 时间：2026年，代表如Hermes Agent、LLM-Wiki等
- 核心理念：解决“静态模型”与“动态世界”矛盾
- 机制：完成任务过程中沉淀经验，通过记忆、反馈循环、自我反思转化为知识或策略
- 目标：“越用越好用”，自动优化提示词、工具策略甚至微调模型
- 意义：Agent从“一次性消耗品”变为“可积累资产”

## 六大核心概念演变

### 1. Prompt：深耦合 → 渐进式加载
- 早期：一个任务一个Agent，System Prompt包含人设、任务、约束、示例等，维护混乱
- 演进：系统指令与动态任务解耦，System Prompt仅保留底层通用指令和行为规范
- 动态内容存入外部文件（如SKILL.md、USER.md、SOUL.md、CLAUDE.md等），通过渐进式披露加载
- 效果：System Prompt稳定纯粹，业务逻辑模块化，降低维护复杂度

### 2. Planning：思维链 → 复杂长程任务
- 早期：依赖CoT线性引导，简单任务尚可，复杂任务易逻辑中断
- 当前：基于模型推理能力提升，具备1) 复杂问题结构化分解为子任务并生成Todo List；2) 多步协同长程推理，动态调整计划；3) 动态实例化子Agent协同完成专项任务
- 驱动力：基座模型推理能力升级，Planning从提示技巧进化为智能决策中枢

### 3. Memory：检索增强 → 文件系统化
- 短期记忆：从简单堆砌转向压缩管理，使用阈值控制、结构化摘要、重点提取，提升token利用率和注意力集中度
- 长期记忆：从向量数据库主导回归文件系统，细分为：
  - 事项型记忆：用户偏好、待办等动态事实，使用MEMORY.md或日志文件存储，易读可控
  - 知识型记忆：受LLM-Wiki等启发，用本地文件系统+Obsidian等构建知识库，结合轻量向量检索（如QMD、SQLite）或企业级向量检索，实现“文件系统化沉淀+向量检索混合管理”

### 4. Tools：Function Call → CLI / Script
- 早期：Function Call封装API，开发成本高，Schema管理复杂；MCP仅优化发现机制，未改变底层逻辑
- 范式转移：
  - CLI原生化：模型预训练掌握Linux命令，零样本调用；支持--help自解释，按需学习；可通过Skill包装新CLI工具
  - Script脚本化：工具逻辑封装为Python等脚本，隐藏API细节，Agent只需调用脚本并传参；Skill内置可执行脚本，赋予Agent处理复杂任务的能力
- 核心：从“人为适配模型”到“利用模型原生能力”

### 5. Workflow：刚性编排 → 动态混合封装
- 早期：硬编码状态机或流水线，强制按步骤执行，保证下限但机械，无法动态调整
- 演变：
  - 逻辑内聚化：任务步骤、约束写入Skill的Markdown文件，模型阅读文档理解完整链路
  - 执行脚本化：精确控制环节用Script编排，复杂流程封装为可复用Skills
- 当前混合策略：成熟标准化子任务封装为Skill，关键主干流程保留Workflow或作为Tool兜底，兼顾灵活性与确定性

### 6. Environment：无状态 → 运行时环境
- 早期：工具调用无状态，无需运行环境
- 当前：Agent需文件读写、代码执行、持久化状态，需要专属Workspace
- 两种形态：
  - 本地个人电脑：灵活，可操作本地文件和应用，但需权限控制
  - 沙箱/云服务器：容器化隔离，安全边界，适合企业生产

## 总结
Agent的宏观模块未变，但每个模块的底层逻辑、数据流转、工程范式发生巨变，即“形未变，神已变”。核心思想是通过工程化手段构建确定性，承载模型不确定性，Agent从“魔法调优”迈向“系统工程”。

## 引用与数据

 今天的Agent依然由Prompt、Planning、Memory、Tools等经典模块组成，这与Lilian Weng早期提出的理论框架并无二致。“形”未变，但“神”已大不同。
这种“通过工程化手段构建确定性，以承载模型不确定性”的核心思想，将是未来很长一段时间内构建高质量Agent的基石。
模型会继续升级、工具会继续变化，框架会持续更新，但这种核心思想将是个基石。
