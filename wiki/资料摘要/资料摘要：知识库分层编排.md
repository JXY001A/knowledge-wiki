> 本文探讨了从RAG到Agent-native知识上下文层的知识库演进路径。文章指出Naive RAG存在无法积累、难以形成全局理解和粒度混乱三大结构性缺陷。通过对比四种主流范式（Naive RAG、LLM Wiki、Graphify、GraphRAG），作者提出了“金字塔”范式，将知识按稳定性和抽象度分为五层并结合知识图谱与角色适配，实现结构化路由检索，旨在让不同角色的AI能找到正确层次、关联和角色的答案。

## 核心要点

- Naive RAG存在三大缺陷：每次从零推导、无法连点成线、粒度混乱，根源在于缺乏结构。
- 四种知识库范式对比：LLM Wiki强调持续编译，Graphify构建代码图谱，GraphRAG利用社区摘要，但均缺少层次感知和角色适配。
- 金字塔范式将知识按稳定性分为L1原则至L5经验五层，并通过7种有向边构建跨层知识图谱。
- 金字塔通过角色-层级访问矩阵和context_budget实现角色感知，为不同角色提供定制化的知识视图。
- 知识保鲜依赖分层审查周期、自动化审计指标和变更驱动的增量同步机制。
- 实验表明金字塔+向量检索的混合方案（Pyramid+RAG）在Hit@3等指标上优于纯Naive RAG。

## 详细笔记

## 一、知识库的根本困境与RAG的天花板

### 1.1 Naive RAG的结构性缺陷

- **缺陷一：每次从零推导**。引用Karpathy的观点：“LLM在每个问题上都从头重新发现知识，没有任何积累。”问一个需要综合5篇文档的问题，LLM必须每次都找到并拼接这5个片段，无中间成果保留。
- **缺陷二：无法连点成线**。Microsoft GraphRAG研究指出两点失败模式：一是当答案需通过共享属性连接分散信息时，平坦的向量检索无能为力；二是无法对大规模语料做全局性的语义理解。
- **缺陷三：粒度混乱**。一个chunk可能是“系统设计原则”也可能是“某个函数的第42-143行实现”。向量空间不区分抽象层次，导致它们可能因共享关键词而同时返回，但服务于完全不同的认知需求。

### 1.2 常见症状

- “搜什么都是那几篇” → 高词频长文档垄断Top-K结果
- “找到了但不是我要的层次” → 想知道“是什么”，返回了“怎么实现”
- “改了一个地方不知道影响什么” → 文档之间没有关联关系
- “新人不知道从哪看起” → 没有阅读路径和导航结构

**共同根源：知识库缺少结构。**向量检索把知识当成“一袋词”，而工程知识是“一棵树”和“一张图”。

## 二、四种知识库范式全景对比

### 范式一：Naive RAG — 平铺向量检索
- **核心思想**：文档 → chunk → embedding → 向量数据库 → 相似度检索。
- **优势**：实现简单，无需预处理，直接可用。
- **局限**：无积累、无关联、无层次、无角色区分。虽可通过metadata filter、rerank、hybrid search等手段弥补，但需要额外工程投入。

### 范式二：LLM Wiki — 持续编译的知识工件
“wiki是一个持续积累的工件。交叉引用已经建好，矛盾已经被标记，综合分析已经反映了所有已读内容。”
- **三层架构**：Raw Sources（人类策展，不可变源文档）→ Wiki（LLM生成并维护的结构化markdown页面）→ Schema（定义结构和工作流的配置文件）。
- **三个核心操作**：**Ingest**（新文档进入后台更新所有相关页面）、**Query**（通过index.md定位后综合回答，好回答可反哺为页面）、**Lint**（定期健康检查矛盾、过期、断裂引用）。
- **为何可行**：Karpathy指出“人类放弃wiki是因为维护负担增长快于价值”，而LLM做摘要、交叉引用、归档的成本远低于人工，但仍有幻觉和内容冲突风险，需人工审核。
- **局限**：页面关联通过手动wikilink维护，无自动关系推断，适合中等规模（~100源文档）。

### 范式三：Graphify — 代码即图谱
- **核心思想**：将代码库、文档、配置等异构资源统一映射为可查询的知识图谱。
- **双管道提取**：**AST管道**（离线，tree-sitter解析提取函数、类等代码实体）与**语义管道**（LLM提取文档中的概念和关系）。
- **三个产出物**：**graph.html**（可视化）、**GRAPH_REPORT.md**（洞察报告，含God Nodes、Surprising Connections、Knowledge Gaps）、**graph.json**（完整图谱数据）。
- **独有能力**：自动发现连接度最高的概念枢纽（God Nodes）、意外的跨模块关联、知识缺口。每条关系有“提取/推断/模糊”三档置信度。
- **局限**：擅长关联分析（“A和B有什么关系”），不擅长直接问答（“这个接口参数是什么”）。

### 范式四：GraphRAG — 图谱增强的检索
- **核心思想**：先构建知识图谱 → Leiden算法社区聚类 → 生成分层社区摘要 → 查询时结合图结构和摘要。
- **两种查询模式**：**Global Search**（利用社区摘要做全局推理）和**Local Search**（从特定实体出发沿边扩展）。
- **核心能力**：通过图结构“连点成线”，通过社区摘要实现“全局理解”，解决了Naive RAG的两个痛点。
- **局限**：构建成本高（大量LLM调用做实体提取），增量更新困难，对源文档质量敏感。

## 三、金字塔：一种新的知识工程范式

**核心创新**：把知识按稳定性和抽象度分为5层，每层服务不同的认知需求和角色，补上了“层次感知+角色适配”这一环。

### 五层分层设计
| 层 | 对应 | 稳定性 |
|---|---|---|
| L1 原则 | SOLID / KISS / YAGNI | 最高（年） |
| L2 架构 | 架构决策记录（ADR） | 高（季度） |
| L3 规范 | 编码标准（ESLint规则） | 中（月） |
| L4 实现 | 代码模板、SDK文档 | 低（周） |
| L5 经验 | 故障复盘、运维日志 | 最低（天） |

### 知识图谱与角色感知
- **7种有向边关联**：governs（原则约束架构）、defines、constrains、implements、validates、feedback（经验反馈改进规范）、cross_ref。支持上溯、下探和反馈环。
- **角色-层级访问矩阵**：架构师看L1+L2，开发者看L2+L3+L4。每个角色有独立的`context_budget`，系统按优先层顺序逐层填充内容直到预算用完。

### 检索机制
- **结构化路由**：先确定去哪层找，再精确定位。所有检索在本地完成，无需网络调用。
- **与向量检索对比**：金字塔通过分层+角色过滤将搜索空间大幅缩小，代码级深度稍弱。**最优组合**：金字塔做分层定位（0 API调用）→ 向量检索补代码级深度（1 API调用）。

### 与其它范式的关系
金字塔是831篇源文档的“索引+摘要+导航图”，在其顶层增加了结构化的路由和导航层。

## 四、同步机制

### 知识腐烂问题
- **三种腐烂形式**：静默过期（接口已变文档未更，危害最大）、层级漂移（架构决策已降级却未挪层）、覆盖盲区（新服务缺少文档）。
- **判断标准**：如果新人按知识库操作后会踩坑，文档就已经腐烂了。

### 保鲜原则
1.  **分层审查周期**：L1年度，L2季度，L3月度，L4/L5每周/每天。
2.  **自动化审计**：设定了**覆盖率**（核心服务/每层是否覆盖）、**新鲜度**、**图谱连通**（无孤立节点）、**层级平衡**四项健康指标。
3.  **变更驱动更新**：绑定到架构评审、Lint规则变更、依赖升级、故障复盘、新人入职提问等工作流事件。

## 五、测评结果与局限

- **实验条件**：831篇源文档，200条QA pair评测集，采用RAGAS框架评估。
- **核心结论**：**Pyramid+RAG混合方案在Hit@3（89.0% vs 约75%）等指标上优于纯Naive RAG**。Knowledge Graph在Hit@1（64.5%）上表现突出。
- **局限性声明**：单评估者、非盲评、评测集为LLM生成、A模式数据为估算、仅在单一团队知识库上测试。

**最终目标**：程序员问一个问题，AI能在3秒内返回正确层次、正确角色、正确关联的答案，而不是5段不相关的文本。

## 引用与数据

- Karpathy: "the LLM is rediscovering knowledge from scratch on every question. There's no accumulation."
- Microsoft GraphRAG 研究指出 baseline RAG "struggles to connect the dots" 及 "performs poorly when being asked to holistically understand summarized semantic concepts over large data collections."
- Karpathy: "Humans abandon wikis because the maintenance burden grows faster than the value."
- 知识库最大的敌人不是“没有内容”，而是“内容过期”。过期的文档比没有文档更危险——因为它给你错误的信心。
- 金字塔思路的核心价值不在于替代任何一种知识库，而是给知识加上结构——让不同角色AI知道该去哪里找、按什么顺序读、给谁看哪些。
